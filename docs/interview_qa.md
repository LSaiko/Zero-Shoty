# Interview Q&A: Zero-Shot Text Classifier

## Q1: What is zero-shot classification and why does BART-MNLI enable it?

Zero-shot classification is the task of assigning labels to text without having seen any labeled training examples for those specific labels at training time. Instead of training a dedicated classifier head for a fixed label set, the model reuses knowledge learned from a different, more general task and applies it to arbitrary, user-supplied labels at inference time. `facebook/bart-large-mnli` enables this because it is a BART sequence-to-sequence model fine-tuned on the Multi-Genre Natural Language Inference (MNLI) dataset, which teaches it to judge whether one sentence (a premise) entails, contradicts, or is neutral toward another sentence (a hypothesis). The zero-shot trick, popularized by Yin et al. (2019), reframes classification as NLI: the input text becomes the premise, and each candidate label is turned into a hypothesis template like "This example is about {label}." The model then scores the entailment probability for each label, and the label set can change freely between calls since the model was never trained on those specific classes — it is only ever solving the general NLI problem.

## Q2: What is Natural Language Inference (NLI) and how does entailment scoring work?

NLI is the task of determining the logical relationship between a premise and a hypothesis: entailment (the hypothesis follows from the premise), contradiction (the hypothesis contradicts the premise), or neutral (neither follows nor contradicts). A model trained on NLI datasets like MNLI learns to output a probability distribution over these three classes for any premise-hypothesis pair. In zero-shot classification, entailment scoring repurposes this three-way output: for each candidate label, the classifier constructs a hypothesis from a template, runs the premise-hypothesis pair through the model, and takes the entailment logit (ignoring contradiction and typically renormalizing against neutral) as the raw signal for "how well does this label describe this text." When multiple labels are compared, the entailment probabilities across all label-hypothesis pairs are normalized (e.g., via softmax) so they form a comparable score distribution, which is what gets returned as the per-label confidence scores.

## Q3: Why does zero-shot classification degrade as label count increases?

As the number of candidate labels grows, several things work against accuracy. First, entailment scores are typically computed independently per label and then normalized across the set, so with many labels the probability mass gets spread thinner, making the true label's score less distinctly separated from the others (lower margin between top candidates). Second, larger label sets are more likely to contain semantically overlapping or near-synonymous labels (e.g., "annoyed" vs. "frustrated"), which genuinely confuses entailment judgments even for a human. Third, the model has a fixed context window and per-pair inference cost — with many labels, more hypothesis templates must be scored, increasing the chance that some label's phrasing is a poor or ambiguous match for the underlying template, introducing noise into the ranking. Finally, MNLI's training distribution is dominated by relatively simple, everyday sentence pairs, so it has no learned prior over your specific domain's fine-grained taxonomy — with more labels, subtle distinctions increasingly fall outside what the model's entailment judgment was ever calibrated to resolve.

## Q4: How would you improve label quality without labeled training data?

Without labeled data, the lever available is label *engineering* rather than model fine-tuning. Concretely: write labels as natural, descriptive phrases rather than single ambiguous words, since the model's entailment judgment is sensitive to how naturally the hypothesis template reads (e.g., "a request for a refund" scores more reliably than "refund"). Deduplicate and merge near-synonymous labels so probability mass is not needlessly split. Keep label sets small and mutually exclusive where possible, since flatter, more orthogonal taxonomies produce cleaner entailment separation. Iterate using the model's own uncertainty signal — flag low-confidence or low-margin predictions (as `is_ambiguous` does) and route those specific cases to a secondary review step, whether that's a human or, as in this project, an LLM advisor that critiques the label set itself based on where the classifier struggled. Finally, running a small held-out set of representative examples and manually eyeballing failure patterns (even 20-30 examples) often reveals systematic label wording issues that can be fixed without any formal training data.

## Q5: What is the difference between zero-shot and few-shot classification?

Zero-shot classification makes predictions with no task-specific examples at all — the model relies purely on generalization from its pretraining or fine-tuning objective (here, NLI) to interpret novel labels it has never explicitly seen paired with this task. Few-shot classification, by contrast, provides the model with a small number of labeled examples (typically embedded directly in a prompt for an LLM, or used for lightweight fine-tuning/calibration in a classical setup) that demonstrate what each label means and how it should be applied. Few-shot generally yields higher accuracy because the examples disambiguate label semantics and calibrate the decision boundary to the specific domain, at the cost of requiring someone to curate representative labeled examples. In this project's architecture, the BART-MNLI classifier is strictly zero-shot (labels are supplied fresh at inference time with no examples), while the Claude advisor could, in principle, be given a few-shot prompt of past ambiguous cases and their resolutions to improve its critique quality — but that's a prompting choice for the advisor role, not a change to the classifier itself.

## Q6: How does Claude's role in this project differ from the BART-MNLI model's role?

BART-MNLI is the classifier: it is a smaller, purpose-fit model that does the actual per-text label scoring, producing a confidence distribution over the candidate labels for every input. It runs locally via the `transformers` pipeline, is comparatively cheap and fast, and its only job is discriminative — outputting numbers, not explanations. Claude, by design in this project, is never the classifier and never scores text against labels directly. It plays a strictly advisory role: it is invoked only when the classifier's output is ambiguous (low top score or a small margin between the top candidates), and even then it receives the text, the label set, and the classifier's scores as context, not the raw classification task. Its job is qualitative critique — flagging that the label set may be poorly defined, that two labels overlap, or that the text is genuinely ambiguous — and it returns suggestions for a human to act on, capped at 300 output tokens to keep it a lightweight, bounded consult rather than a parallel classification path. This separation keeps the expensive, general-purpose LLM out of the hot path and reserves it for the comparatively rare cases where nuanced judgment adds value over a fast statistical model.

## Q7: If you were to productionize this, what would change?

Several things would need to change for production readiness. Performance: the BART-MNLI pipeline would need batching, GPU placement, and likely quantization or a distilled variant to meet latency SLAs, plus the model should be loaded once at process startup rather than per-request. Reliability: the Claude advisor calls need real retry/backoff logic, timeouts, and circuit-breaking so an Anthropic API outage never blocks the primary classification path — the advisor must stay a non-blocking, best-effort enhancement. Observability: structured logging (already using `logging` instead of `print`) would be extended with request tracing, latency histograms, and score-distribution monitoring to detect label-set drift or systematic ambiguity spikes over time. Cost control: since the advisor is only invoked on ambiguous predictions, that gating logic itself becomes a cost lever worth tuning and monitoring (advisor invocation rate as a first-class metric). Testing: the current stubs would be backed by real unit tests with mocked HF/Claude calls plus a small golden-set integration test suite to catch regressions in label wording or threshold tuning. Finally, security and compliance: API keys via a secrets manager rather than environment variables, PII scrubbing before any text is sent to Claude, and a clear data-retention policy for the third-party API calls.
